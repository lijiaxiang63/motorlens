## Verification without a camera

Everything downstream of a `FrameSource` is testable with synthetic data; use this before asking the user to camera-test:

- Unit tests validate the full metrics pipeline against `src/replay/synthetic.ts`, whose generators return exact ground truth (`{frames, truth}`) â€” event times, amplitudes, ITIs are scheduled, not sampled.
- Headless browser: `/?source=synthetic&preset=tap-2hz|tap-decrement|tap-hesitant|tap-slow|fist-1p5hz|angles-sweep&speed=4`, click through the flow, then assert on `window.__lastReport` (SessionReport of the last results screen) and `window.__ctx` (the AppContext; `__ctx.source.kind` tells you which source is live). Presets run 16 s so a positioning + countdown + 10 s recording fits before the pattern loops; the record screen calls `source.restart()` on mount.
- Expected preset results: tap-2hz â†’ 20 taps / 2.00 Hz / ~0 decrement; tap-decrement â†’ ~30% (25â€“35); fist-1p5hz â†’ 15 cycles / 1.50 Hz; tap-hesitant â†’ 2 hesitations; angles-sweep â†’ per-joint ROM equal to the sweep maxima in `src/replay/presets.ts`.
- **Ready-made CDP drivers in `scripts/e2e/`** (no Playwright; plain Node â‰¥22 + local Chrome, or attach to Electron via `--ws`): `preset-flow.mjs` runs preset flows and dumps `__lastReport` JSONs; `compare-reports.mjs` diffs two report dirs modulo the absolute time base (the parity gate â€” raw frame times are exported rounded, so byte-diffing raw JSONs is the wrong assertion); `subject-flow.mjs`, `import-flow.mjs`, `video-flow.mjs` cover the flows below; `electron-flow.mjs` runs presets inside Electron against the built `app://bundle` (run `npm run build && npm run build:electron` first; kill stray Electron instances first â€” a leftover instance makes new launches misbehave). Start `npm run dev` yourself before the browser-based flows.
- Camera-only code (`src/camera/`, `src/tracking/cameraSource.ts`) is lazy-imported in `main.tsx` and never loads in synthetic mode â€” it cannot be exercised headlessly; changes there need a real-camera check by the user. The same lazy-import rule covers `src/tracking/videoFile.ts` (dynamic import inside `ui/screens/VideoReviewScreen.tsx`): MediaPipe must stay out of the main bundle (check `vite build` output: `landmarks`/`cameraSource`/`videoFile` must be separate chunks).
- Subject flow headlessly: home â†’ "Open subjects" â†’ create a subject â†’ checklist "Start" runs recordâ†’results with a subject context; the results screen auto-saves to IndexedDB (assert via `indexedDB.open('motorlens')`, stores `subjects`/`results`/`videos`/`settings`) and `window.__lastReport.subject/source` are set. **Synthetic presets emit a right hand only** â€” the checklist's left-hand rows can never pass the positioning gate headlessly, so drive the "â€¦ â€” Right hand" rows. "Export all (ZIP)" sets `window.__lastExport` (Blob) before triggering the download. The upload path can be smoke-tested by recording a canvas `captureStream()` into a webm File and navigating `__ctx.navigate({name:'videoReview', subject, file})` â€” no hands detected means 0 auto-segments, but processing, review UI, manual segments, analysis, and persistence all execute (and MediaRecorder webm exercises the Infinity-duration workaround).
- MediaRecorder capture (`src/capture/videoRecorder.ts`) is inert on synthetic sources (`source.video` is null); it needs a real-camera check.

## Architecture

**The `FrameSource` seam.** Camera, synthetic, and replayed-JSON sources all emit identical `LandmarkFrame`s (`src/types.ts`) and implement `FrameSource`. Nothing downstream knows which is which. `handedness` on a frame is already mirror-corrected physical handedness.

**Pure analysis core.** `src/signal/` (filters, stats, scipy-equivalent peak detection) and `src/metrics/` are DOM-free pure functions. The offline pipeline (`metrics/cycleTest.ts`) runs on the full recording: segment split at >300 ms tracking gaps â†’ zero-phase forward-backward EMA (deliberately non-causal: no peak lag/attenuation bias; live charts use causal `LiveEma` for display only) â†’ adaptive-prominence peak/valley detection â†’ shared cycle engine (`metrics/cycles.ts`: an event = valley with preceding peak, after alternation cleanup) â†’ aggregates (decrement via OLS over event index, rhythm from ITIs within segments). The live tap counter re-runs this same offline pipeline on the accumulated buffer every 250 ms â€” there is intentionally no separate streaming detector, so live and final counts can't diverge.

**Two coordinate systems, two jobs.** Movement signals and normalization use MediaPipe *world* landmarks (metric 3-D): rigid rotation preserves 3-D distances, so palm tilt during taps/clenches cannot fake amplitude changes or decrement. Signals are divided by world hand scale |W0âˆ’W9| ("hand units" â‰ˆ palm lengths; Ã—`cmPerUnit` for cm). Projected image-space distances are used only for framing gates (too far/close) and the positioning-stability quality metric. `taps.test.ts` has a pitch-wobble test enforcing this invariant â€” don't move signal math back to image space.

**Frame-time-driven protocol.** `protocol/testSession.ts` is a state machine driven solely by `onFrame(f)` and frame timestamps â€” no wall-clock reads â€” so it behaves identically at any replay `speed` and is unit-testable.

**React UI (React 19 + Tailwind v4 + owned shadcn-style components).** `src/main.tsx` bootstraps; `ui/root.tsx` is the screen switch; `ui/nav.tsx` is a typed in-memory state router â€” `ScreenRequest` payloads carry live objects (`TestDefinition` functions, `File`, `Blob`, frames), so nothing serializes and there are no URLs. Screens are keyed by `navCount` so every navigation remounts (effect teardown replaces the old `destroy()` contract). `window.__ctx = {source, navigate, replaceSource}` keeps a **stable object identity** with a mutable `source` property (assigned once in a NavProvider effect); `__lastReport` is set by `ResultsScreen`, `__lastExport` by `SubjectsScreen` â€” the headless harness depends on all three. StrictMode is ON in dev: side effects that must fire once (MediaRecorder start, results auto-save, done-navigation) key off refs tracking phase *transitions*, not effect bodies. Frame data (30â€“120 Hz) never flows through React props/state: `PreviewPanel` and screens subscribe to the source directly (`useFrameSubscription`); charts take imperative `push()` handles (`ui/charts/charts.tsx` wrapping `ui/charts/uplotCore.ts`); only the small stage panel re-renders from `TestSession` phase via `useSyncExternalStore`. All colors live in `ui/tokens.css` (semantic vars per theme, mapped via `@theme inline`; chart tokens validated for CVD/contrast per mode); canvases read them via `getComputedStyle` at mount and are recreated on theme switch (`useTheme().resolved`). Theme pref (`light`/`dark`/`system`, default `system`) persists to the IDB `settings` store with a localStorage mirror (`motorlens.theme`) read by an inline `index.html` script before first paint. Screen inventory: `HomeScreen` (quick test), `RecordScreen`, `ResultsScreen`, `SubjectsScreen`, `SubjectScreen` (battery workbench), `VideoReviewScreen`, `MonitorScreen`, `SettingsScreen`; shared pieces in `ui/components/` (owned shadcn-style primitives under `components/ui/`), hooks in `ui/hooks/`, shell in `ui/shell/Shell.tsx`.

**Report round-trip as regression harness.** Results build a `SessionReport` (schemaVersion 1) with raw frames rounded on export (coords to 4 dp, `t` to 0.1 ms); importing an exported JSON recomputes metrics from `raw.frames`. The locked contract (`report/export.test.ts`) is: **count exact, frequency to 1e-6** â€” velocity-family metrics and near-zero decrement percentages recompute to ~0.3% / a few tenths of a point because derivatives amplify the rounding; displayed values (1 dp / 0 dp) reproduce exactly. Don't byte-diff reports across runs either: absolute frame timestamps shift with page-load timing (`compare-reports.mjs` rebases to the first frame before comparing). The optional `subject`/`source` fields are additive â€” `parseSessionJson` only validates schemaVersion/app/frames, and `buildSessionReport` spreads them only when present, so pre-subject reports stay byte-identical.

**Subjects & persistence.** `src/store/db.ts` (promise IDB wrapper) + `src/store/subjects.ts` (Subject / StoredResult / StoredVideo, cascade delete, video refcount GC, saveVideo setting). Results auto-save on the results screen only when a subject context is threaded through `ScreenRequest` (`subjectCtx` on record â†’ `subject` on results); the quick-test flow never touches IndexedDB. Live captures store one video per result (`live_<id>`); all segments of one uploaded file share `upload_<uuid>`. Batch export (`report/batch.ts`, fflate `zipSync`, videos at level 0) writes `summary.csv` (`report/csv.ts`, UTF-8 BOM+CRLF, column set locked by tests) plus one folder per subject; it assembles in RAM â€” fine for tens of subjects, streaming is future work.

**Uploaded-video analysis.** `tracking/videoFile.ts` seek-steps a file at `VIDEO_PROC_FPS` with its OWN HandLandmarker (detectForVideo timestamps are per-instance monotonic; the live camera is unaffected) and emits mediaTime-based frames â€” correct for files, and exactly the player-timeline coordinate the review screen needs (the wall-clock rule below is getUserMedia-specific). `metrics/segments.ts` (pure, ground-truth-tested like the metrics) auto-segments: majority-vote handedness smoothing â†’ presence runs â†’ sliding-window tap/fist classification â†’ merge. The classifier checks **fist first** via `restRaw` (middle/ring/pinkyâ†’wrist; index excluded) because thumbâ€“index distance oscillates during fists too. `sliceFrames` converts other-hand frames to blank frames instead of dropping them: the time base survives, detectionRate means "target hand visible", and >`MAX_GAP_MS` holes still split analysis segments. `ui/screens/VideoReviewScreen.tsx` renders the correction timeline (`ui/segmentTimeline.ts`, imperative canvas) and saves one result per confirmed segment.

**Electron shell (Phase 0).** `electron/main.ts` serves the built renderer from the privileged `app://bundle` origin (protocol handler: MIME table incl. wasm, HEAD support for the MediaPipe probe, SPA fallback preserving query strings) â€” that origin is the IndexedDB identity and must never change. `electron/preload.ts` exposes `window.motorlens` (native save/open dialogs, appInfo, update IPC, sync `flags`); `src/platform.ts` is the only renderer file aware of it (mirrors the bridge's types without importing `electron/shared.ts`, so it stays usable from the browser build). Camera permission is granted via `setPermissionRequestHandler` + `askForMediaAccess`. Synthetic argv passthrough (`--source=synthetic --preset=â€¦ --speed=â€¦`) enables the headless flows in the shell. Code signing stays disabled (`identity: null`) â€” no Apple Developer ID / EV certificate yet; see the auto-update bullet below for how that constrains macOS updates.

**Window chrome (Phase 5).** macOS uses `titleBarStyle: 'hiddenInset'` with an explicit `trafficLightPosition: {x:18,y:18}`; the Sidebar logo header in `Shell.tsx` reserves a matching `pt-[52px]` band (mac-desktop only, via a module-scope `macDesktop = isDesktop() && navigator.platform.startsWith('Mac')`) and is itself the drag region â€” **keep these two coupled** if either changes, or the traffic lights overlap the logo again (that was the original bug). macOS also gets a `vibrancy:'sidebar'` material: `main.ts` sets `transparent: true` + a fully-transparent `backgroundColor` (Electron only honors the alpha byte in `backgroundColor` when `transparent` is set â€” a real gotcha, cost a debugging pass) and passes `--motorlens-vibrancy` via `webPreferences.additionalArguments`; preload exposes it synchronously as `window.motorlens.flags.vibrancy` (no IPC round-trip) so index.html's pre-paint script can add a `.vibrancy` class to `<html>` before first render. `tokens.css`'s `.vibrancy` rules are deliberately **unlayered** (like the existing `@media print` block) so they beat Tailwind's layered utility classes regardless of selector specificity; the content column (`.app-content`) keeps an explicit opaque `bg-background` so charts/camera preview are pixel-identical to non-vibrancy builds. The `show:false` + `ready-to-show` gate is now the *only* anti-flash protection (backgroundColor can't do it once it's transparent) â€” don't drop it.

**App icon (Phase 5).** `scripts/make-icons.mjs` renders one hand-authored SVG (a hand-landmark glyph â€” wrist + 5 joint-dot chains, echoing the MediaPipe hand graph) at multiple sizes via headless Chrome (reusing `scripts/e2e/cdp.mjs`, no new native deps), then packs `build/icon.icns` via macOS's `iconutil` and hand-rolls `build/icon.ico` (zero-dep: ICO's PNG-compressed entry format needs no BMP encoding). Outputs (`build/icon.{png,icns,ico}`, `public/{icon.png,favicon.svg}`) are **committed**, not generated in CI â€” run `npm run icons` and commit the results when the design changes. `electron/main.ts` sets `icon:` on non-darwin only (macOS reads `build/icon.icns` from the app bundle via electron-builder's `mac.icon` config).

**Auto-update (Phase 5).** `electron/updater.ts` picks a delivery path once at startup via `fullUpdater = win32 || (darwin && MAC_FULL_UPDATER)` (`MAC_FULL_UPDATER = false` today â€” flip it once the mac build is code-signed + notarized). Windows (and mac once signed) gets the real `electron-updater` flow against the GitHub Releases feed electron-builder embeds via the `publish` config in `package.json`; **`publish` must exist before any Windows build** or electron-updater throws at init (missing `app-update.yml`). Unsigned macOS today can't self-update at all â€” Squirrel.Mac validates a code signature before installing â€” so it instead polls the GitHub "latest release" API directly and surfaces a `shell.openExternal` deep link. Both paths share one `UpdateStatus` push (`electron/shared.ts`) over an IPC channel (`motorlens:update-event`) driven by `electron/updater.ts`'s `push()`; the renderer's `SettingsScreen`'s `UpdateRow` and `Shell.tsx`'s one-shot toast both subscribe via `onUpdateEvent`. **Register `initUpdater()` before `createWindow()`**, not after â€” the renderer can call `updateCheck()` as soon as `window.motorlens` exists (synchronously, via preload), which races an IPC handler registered later (this was a real bug, caught by the headless test below). Dev builds no-op to `state:'dev'` unless `MOTORLENS_UPDATE_FEED` (generic feed, full-updater path) or `MOTORLENS_RELEASE_API` (GitHub API override, check-only path) is set â€” that's how the whole state machine is verified headlessly without publishing a release: spin up a tiny local HTTP server returning `{tag_name, html_url}`, launch `npx electron .` with the env var set, and `window.motorlens.updateCheck()` over CDP.

**Version single-sourcing.** `APP_VERSION` (`src/config.ts`) reads the build-time global `__APP_VERSION__`, injected from `package.json`'s `version` by a `define` in **both** `vite.config.ts` and `vitest.config.ts` (they're separate configs/processes â€” a define in only one leaves the other's `import`s of `APP_VERSION` throwing `ReferenceError` at module evaluation). `src/vite-env.d.ts` holds the ambient `declare const __APP_VERSION__: string`. Bumping the version is `npm version <bump>` (updates `package.json`, which both the renderer's `APP_VERSION` and electron-builder read) â€” restart `npm run dev` afterward, since Vite's `define` is evaluated at server startup, not per-request.

## Hard-won invariants (each fixed a real-hardware bug â€” do not "simplify" away)

- **Camera frame timestamps are wall clock** (the rVFC callback's `now`), never `video.mediaTime`: mediaTime stalls for getUserMedia streams on some stacks, and the strictly-increasing fallback degrades to +1 ms/frame â€” countdown/test timing then crawls ~30Ã— slow.
- **`SWAP_RAW_HANDEDNESS = false`** in `src/config.ts` was verified empirically on the user's camera (the MediaPipe-docs-suggested swap made every right hand read as left). Flip only with a real-camera re-test.
- **The frame chain must survive exceptions**: `frameClock.ts` try/catches the callback, `cameraSource.ts` turns `detectForVideo` failures into no-hand frames, and both sources isolate each subscriber. One throwing consumer previously froze the whole app.
- **`replaySource.ts` does wall-clock catch-up batching**: hidden tabs throttle `setTimeout` to 1 Hz; emitting only one frame per tick stalls playback. It also inserts a >`MAX_GAP_MS` time gap at loop/restart boundaries so analyses segment there.
- **The camera `<video>` element doesn't exist until the camera opens** â€” `ui/PreviewPanel.tsx` late-attaches it in its rAF loop into a dedicated host div that React renders empty (JSX must never own that element â€” reconciliation would detach it), and re-`play()`s it whenever a DOM move paused it. Screens created before camera-ready would otherwise show skeleton-only forever.
- **uPlot sizing**: charts mount on an inner 100%-width div and resize only when width actually changes (`ui/charts/uplotCore.ts`) â€” observing the padded panel directly causes a ResizeObserverâ†”setSize feedback loop inside grid tracks (`min-width: 0` on grid children matters too; the `min-w-0-children` utility exists for this).
- **StrictMode-safe one-shot side effects**: MediaRecorder start / results auto-save / done-navigation are guarded by refs (`startedRef`, `savedRef`, `finishedRef`) that survive StrictMode's double-invoked effects â€” don't move them into effect-local state.
- Recording never aborts on tracking loss; gaps become segments and quality metrics (`droppedIntervals`, detection rate). Positioning gates only guard entry into the countdown.

All tunables (test durations, smoothing cutoffs, peak thresholds, gate limits, asset URLs) live in `src/config.ts`. When adding a metric or test type, extend `TestDefinition` in `protocol/definitions.ts` and add a ground-truth generator + suite mirroring `metrics/taps.test.ts`.
