## Verification without a camera

Everything downstream of a `FrameSource` is testable with synthetic data; use this before asking the user to camera-test:

- Unit tests validate the full metrics pipeline against `src/replay/synthetic.ts`, whose generators return exact ground truth (`{frames, truth}`) — event times, amplitudes, ITIs are scheduled, not sampled.
- Headless browser: `/?source=synthetic&preset=tap-2hz|tap-decrement|tap-hesitant|tap-slow|fist-1p5hz|angles-sweep&speed=4`, click through the flow, then assert on `window.__lastReport` (SessionReport of the last results screen) and `window.__ctx` (the AppContext; `__ctx.source.kind` tells you which source is live). Presets run 16 s so a positioning + countdown + 10 s recording fits before the pattern loops; the record screen calls `source.restart()` on mount.
- Expected preset results: tap-2hz → 20 taps / 2.00 Hz / ~0 decrement; tap-decrement → ~30% (25–35); fist-1p5hz → 15 cycles / 1.50 Hz; tap-hesitant → 2 hesitations; angles-sweep → per-joint ROM equal to the sweep maxima in `src/replay/presets.ts`.
- **Ready-made CDP drivers in `scripts/e2e/`** (no Playwright; plain Node ≥22 + local Chrome, or attach to Electron via `--ws`): `preset-flow.mjs` runs preset flows and dumps `__lastReport` JSONs; `compare-reports.mjs` diffs two report dirs modulo the absolute time base (the parity gate — raw frame times are exported rounded, so byte-diffing raw JSONs is the wrong assertion); `subject-flow.mjs`, `import-flow.mjs`, `video-flow.mjs` cover the flows below; `electron-flow.mjs` runs presets inside Electron against the built `app://bundle` (run `npm run build && npm run build:electron` first; kill stray Electron instances first — a leftover instance makes new launches misbehave). Start `npm run dev` yourself before the browser-based flows.
- Camera-only code (`src/camera/`, `src/tracking/cameraSource.ts`) is lazy-imported in `main.tsx` and never loads in synthetic mode — it cannot be exercised headlessly; changes there need a real-camera check by the user. The same lazy-import rule covers `src/tracking/videoFile.ts` (dynamic import inside `ui/screens/VideoReviewScreen.tsx`): MediaPipe must stay out of the main bundle (check `vite build` output: `landmarks`/`cameraSource`/`videoFile` must be separate chunks).
- Subject flow headlessly: home → "Open subjects" → create a subject → checklist "Start" runs record→results with a subject context; the results screen auto-saves to IndexedDB (assert via `indexedDB.open('motorlens')`, stores `subjects`/`results`/`videos`/`settings`) and `window.__lastReport.subject/source` are set. **Synthetic presets emit a right hand only** — the checklist's left-hand rows can never pass the positioning gate headlessly, so drive the "… — Right hand" rows. "Export all (ZIP)" sets `window.__lastExport` (Blob) before triggering the download. The upload path can be smoke-tested by recording a canvas `captureStream()` into a webm File and navigating `__ctx.navigate({name:'videoReview', subject, file})` — no hands detected means 0 auto-segments, but processing, review UI, manual segments, analysis, and persistence all execute (and MediaRecorder webm exercises the Infinity-duration workaround).
- MediaRecorder capture (`src/capture/videoRecorder.ts`) is inert on synthetic sources (`source.video` is null); it needs a real-camera check.

## Architecture

**The `FrameSource` seam.** Camera, synthetic, and replayed-JSON sources all emit identical `LandmarkFrame`s (`src/types.ts`) and implement `FrameSource`. Nothing downstream knows which is which. `handedness` on a frame is already mirror-corrected physical handedness.

**Pure analysis core.** `src/signal/` (filters, stats, scipy-equivalent peak detection) and `src/metrics/` are DOM-free pure functions. The offline pipeline (`metrics/cycleTest.ts`) runs on the full recording: segment split at >300 ms tracking gaps → zero-phase forward-backward EMA (deliberately non-causal: no peak lag/attenuation bias; live charts use causal `LiveEma` for display only) → adaptive-prominence peak/valley detection → shared cycle engine (`metrics/cycles.ts`: an event = valley with preceding peak, after alternation cleanup) → aggregates (decrement via OLS over event index, rhythm from ITIs within segments). The live tap counter re-runs this same offline pipeline on the accumulated buffer every 250 ms — there is intentionally no separate streaming detector, so live and final counts can't diverge.

**Two coordinate systems, two jobs.** Movement signals and normalization use MediaPipe *world* landmarks (metric 3-D): rigid rotation preserves 3-D distances, so palm tilt during taps/clenches cannot fake amplitude changes or decrement. Signals are divided by world hand scale |W0−W9| ("hand units" ≈ palm lengths; ×`cmPerUnit` for cm). Projected image-space distances are used only for framing gates (too far/close) and the positioning-stability quality metric. `taps.test.ts` has a pitch-wobble test enforcing this invariant — don't move signal math back to image space.

**Frame-time-driven protocol.** `protocol/testSession.ts` is a state machine driven solely by `onFrame(f)` and frame timestamps — no wall-clock reads — so it behaves identically at any replay `speed` and is unit-testable.

**React UI (React 19 + Tailwind v4 + owned shadcn-style components).** `src/main.tsx` bootstraps; `ui/root.tsx` is the screen switch; `ui/nav.tsx` is a typed in-memory state router — `ScreenRequest` payloads carry live objects (`TestDefinition` functions, `File`, `Blob`, frames), so nothing serializes and there are no URLs. Screens are keyed by `navCount` so every navigation remounts (effect teardown replaces the old `destroy()` contract). `window.__ctx = {source, navigate, replaceSource}` keeps a **stable object identity** with a mutable `source` property (assigned once in a NavProvider effect); `__lastReport` is set by `ResultsScreen`, `__lastExport` by `SubjectsScreen` — the headless harness depends on all three. StrictMode is ON in dev: side effects that must fire once (MediaRecorder start, results auto-save, done-navigation) key off refs tracking phase *transitions*, not effect bodies. Frame data (30–120 Hz) never flows through React props/state: `PreviewPanel` and screens subscribe to the source directly (`useFrameSubscription`); charts take imperative `push()` handles (`ui/charts/charts.tsx` wrapping `ui/charts/uplotCore.ts`); only the small stage panel re-renders from `TestSession` phase via `useSyncExternalStore`. All colors live in `ui/tokens.css` (semantic vars per theme, mapped via `@theme inline`; chart tokens validated for CVD/contrast per mode); canvases read them via `getComputedStyle` at mount and are recreated on theme switch (`useTheme().resolved`). Theme pref (`light`/`dark`/`system`, default `system`) persists to the IDB `settings` store with a localStorage mirror (`motorlens.theme`) read by an inline `index.html` script before first paint. Screen inventory: `HomeScreen` (quick test), `RecordScreen`, `ResultsScreen`, `SubjectsScreen`, `SubjectScreen` (battery workbench), `VideoReviewScreen`, `MonitorScreen`, `SettingsScreen`; shared pieces in `ui/components/` (owned shadcn-style primitives under `components/ui/`), hooks in `ui/hooks/`, shell in `ui/shell/Shell.tsx`.

**Report round-trip as regression harness.** Results build a `SessionReport` (schemaVersion 1) with raw frames rounded on export (coords to 4 dp, `t` to 0.1 ms); importing an exported JSON recomputes metrics from `raw.frames`. The locked contract (`report/export.test.ts`) is: **count exact, frequency to 1e-6** — velocity-family metrics and near-zero decrement percentages recompute to ~0.3% / a few tenths of a point because derivatives amplify the rounding; displayed values (1 dp / 0 dp) reproduce exactly. Don't byte-diff reports across runs either: absolute frame timestamps shift with page-load timing (`compare-reports.mjs` rebases to the first frame before comparing). The optional `subject`/`source` fields are additive — `parseSessionJson` only validates schemaVersion/app/frames, and `buildSessionReport` spreads them only when present, so pre-subject reports stay byte-identical.

**Subjects & persistence.** `src/store/db.ts` (promise IDB wrapper) + `src/store/subjects.ts` (Subject / StoredResult / StoredVideo, cascade delete, video refcount GC, saveVideo setting). Results auto-save on the results screen only when a subject context is threaded through `ScreenRequest` (`subjectCtx` on record → `subject` on results); the quick-test flow never touches IndexedDB. Live captures store one video per result (`live_<id>`); all segments of one uploaded file share `upload_<uuid>`. Batch export (`report/batch.ts`, fflate `zipSync`, videos at level 0) writes `summary.csv` (`report/csv.ts`, UTF-8 BOM+CRLF, column set locked by tests) plus one folder per subject; it assembles in RAM — fine for tens of subjects, streaming is future work.

**Uploaded-video analysis.** `tracking/videoFile.ts` seek-steps a file at `VIDEO_PROC_FPS` with its OWN HandLandmarker (detectForVideo timestamps are per-instance monotonic; the live camera is unaffected) and emits mediaTime-based frames — correct for files, and exactly the player-timeline coordinate the review screen needs (the wall-clock rule below is getUserMedia-specific). `metrics/segments.ts` (pure, ground-truth-tested like the metrics) auto-segments: majority-vote handedness smoothing → presence runs → sliding-window tap/fist classification → merge. The classifier checks **fist first** via `restRaw` (middle/ring/pinky→wrist; index excluded) because thumb–index distance oscillates during fists too. `sliceFrames` converts other-hand frames to blank frames instead of dropping them: the time base survives, detectionRate means "target hand visible", and >`MAX_GAP_MS` holes still split analysis segments. `ui/screens/VideoReviewScreen.tsx` renders the correction timeline (`ui/segmentTimeline.ts`, imperative canvas) and saves one result per confirmed segment.

**Electron shell (Phase 0).** `electron/main.ts` serves the built renderer from the privileged `app://bundle` origin (protocol handler: MIME table incl. wasm, HEAD support for the MediaPipe probe, SPA fallback preserving query strings) — that origin is the IndexedDB identity and must never change. `electron/preload.ts` exposes `window.motorlens` (native save/open dialogs, appInfo); `src/platform.ts` is the only renderer file aware of it. Camera permission is granted via `setPermissionRequestHandler` + `askForMediaAccess`. Synthetic argv passthrough (`--source=synthetic --preset=… --speed=…`) enables the headless flows in the shell. Code signing stays disabled (`identity: null`) until Phase 5.

## Hard-won invariants (each fixed a real-hardware bug — do not "simplify" away)

- **Camera frame timestamps are wall clock** (the rVFC callback's `now`), never `video.mediaTime`: mediaTime stalls for getUserMedia streams on some stacks, and the strictly-increasing fallback degrades to +1 ms/frame — countdown/test timing then crawls ~30× slow.
- **`SWAP_RAW_HANDEDNESS = false`** in `src/config.ts` was verified empirically on the user's camera (the MediaPipe-docs-suggested swap made every right hand read as left). Flip only with a real-camera re-test.
- **The frame chain must survive exceptions**: `frameClock.ts` try/catches the callback, `cameraSource.ts` turns `detectForVideo` failures into no-hand frames, and both sources isolate each subscriber. One throwing consumer previously froze the whole app.
- **`replaySource.ts` does wall-clock catch-up batching**: hidden tabs throttle `setTimeout` to 1 Hz; emitting only one frame per tick stalls playback. It also inserts a >`MAX_GAP_MS` time gap at loop/restart boundaries so analyses segment there.
- **The camera `<video>` element doesn't exist until the camera opens** — `ui/PreviewPanel.tsx` late-attaches it in its rAF loop into a dedicated host div that React renders empty (JSX must never own that element — reconciliation would detach it), and re-`play()`s it whenever a DOM move paused it. Screens created before camera-ready would otherwise show skeleton-only forever.
- **uPlot sizing**: charts mount on an inner 100%-width div and resize only when width actually changes (`ui/charts/uplotCore.ts`) — observing the padded panel directly causes a ResizeObserver↔setSize feedback loop inside grid tracks (`min-width: 0` on grid children matters too; the `min-w-0-children` utility exists for this).
- **StrictMode-safe one-shot side effects**: MediaRecorder start / results auto-save / done-navigation are guarded by refs (`startedRef`, `savedRef`, `finishedRef`) that survive StrictMode's double-invoked effects — don't move them into effect-local state.
- Recording never aborts on tracking loss; gaps become segments and quality metrics (`droppedIntervals`, detection rate). Positioning gates only guard entry into the countdown.

All tunables (test durations, smoothing cutoffs, peak thresholds, gate limits, asset URLs) live in `src/config.ts`. When adding a metric or test type, extend `TestDefinition` in `protocol/definitions.ts` and add a ground-truth generator + suite mirroring `metrics/taps.test.ts`.
